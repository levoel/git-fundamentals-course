# Шаблон для ответов автора на review

Каждый комментарий reviewer-а должен получить ответ. У ответа есть три формы:

## 1. `agree` — соглашаюсь, фикшу

Используешь, когда reviewer прав. Не оставляй формат «agreed, will fix» без действия — после ответа сразу делай коммит, который фиксит проблему, и ссылайся на него.

```
### agree

Согласен. Сейчас исправлю — заменю f-string на параметризованный запрос:

```python
df = pd.read_sql(text(query), engine, params={"months": months})
```

Сделаю в commit <sha-будущего-фикса>.
```

После того, как коммит появился — добавь ссылку:

```
Fixed in c1a2b3d.
```

## 2. `discuss` — есть нюанс, нужно обсудить

Используешь, когда не уверен, что именно reviewer имеет в виду, или у тебя есть аргументы за и против. Цель — собрать дополнительный контекст.

```
### discuss

В целом согласен с идеей. Но если мы вынесем расчёт retention в SQL, на BigQuery каждый запрос будет 30+ секунд (проверял на похожей таблице в прошлом квартале).

С другой стороны, если запрос будет один раз в день из Airflow — это ок.

Какой use case ты держишь в голове — ad-hoc дебаг из Jupyter, или batch-расчёт раз в сутки?
```

## 3. `push back` — не согласен, привожу аргументы

Используешь, когда уверен, что твоё решение правильное. Самое сложное — потому что важно не скатиться в эмоции.

**Правила push back:**

- Один факт / бенчмарк / ссылка на доку — обязательно.
- Не «я думаю», а «я измерил» или «в документации написано».
- Готовность к компромиссу — если reviewer найдёт контраргумент, ты обновляешь позицию.

```
### push back

Я думал об этом. Здесь pandas использован осознанно — на этапе расчёта `retention_rate` (`df["active_users"] / df["base"]`) мы делим на per-cohort baseline. На SQL это требует window function с PARTITION BY cohort_month, что на PostgreSQL 14 работает корректно, но даёт примерно тот же план выполнения.

Замерил на синтетических данных (100k cohorts × 12 periods):
- SQL window: 4.2s
- Pandas: 3.8s + 50MB RAM

Разница не значимая. Pandas-версия читабельнее и тестируется юнит-тестами без БД.

Готов перенести в SQL, если есть кейс, когда pandas не справится — например, если когорт станет 10M+.
```

## Формат ответа

В файле `my-response.md` структура:

```markdown
# Response to review comments

## Re: Comment 1 — SQL injection via f-string

### agree | discuss | push back

<2-5 строк ответа>

---

## Re: Comment 2 — hardcoded credentials

### agree | discuss | push back

<2-5 строк ответа>

---
```

И так на каждый из 4 комментариев.

## Чего избегать в ответах

| Антипаттерн | Почему плохо |
|---|---|
| «ок» | Нет конкретики. Reviewer не знает, фиксишь ты или нет. |
| «done» без коммита | Слова без действия. Покажи sha. |
| «но раньше это работало» | Не аргумент. Если код небезопасный сейчас — его надо чинить сейчас. |
| «не работает только у тебя» | Перенос ответственности. Воспроизведи у себя сначала. |
| Эмоциональный ответ без перерыва | Если задело — отложи на 30 минут. Перечитай. Часто звучит уже не так. |
| Полное игнорирование комментария | Худший паттерн. Reviewer перестанет тебе ревьюить. |

## Tone

Reviewer тратит своё время на твой код. Спасибо стоит сказать — особенно если комментарий помог найти реальный bug. «Спасибо, действительно проглядел» **дешевле**, чем долгое объяснение, почему ты «вообще-то знал об этом».

Push back — нормально. Слепое «agree» на всё — хуже, чем аргументированное «не согласен». Reviewer тоже учится, когда ты обоснованно отстоял позицию.
