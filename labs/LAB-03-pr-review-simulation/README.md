# LAB-03: PR Review Simulation — обе стороны барикад

**Время:** ~1.5 часа
**Уровень:** Junior
**Связан с модулями:** 11 (Pull Requests), 12 (Git workflows)

## Цель

Code review — самая частая активность Junior DE после написания кода. И самая болезненная: «как отказать тимлиду в его PR?», «как ответить на придирку, которая не по делу?», «что значит nit vs blocker?».

В этой лабе ты пройдёшь обе роли:

**Часть 1 (Reviewer):** тебе дан PR коллеги с новой dbt-моделью. В нём 5 проблем разной серьёзности — твоя задача найти их и оставить **конструктивные** комментарии в правильном формате.

**Часть 2 (Author):** ты получил review на свой код. Твоя задача — корректно ответить на каждый комментарий (где-то согласиться и сразу пофиксить, где-то аргументированно отстоять решение).

После лабы у тебя будет привычка различать `blocker:` / `nit:` / `question:` и писать review, которые не обижают людей, но реально улучшают код.

## Часть 1. Ты — Reviewer

### Контекст

Твой коллега Максим открыл PR в репозиторий аналитической команды:

> **PR #142: feat: add fct_customer_orders dbt model**
>
> *Adds new fact table fct_customer_orders aggregating order totals by customer. Will be used by Marketing dashboard.*

В папке `teammate-pr/` лежит:

- `models/fct_customer_orders.sql` — собственно код dbt-модели.
- `schema.yml` — описание модели и тестов.
- `PR_DESCRIPTION.md` — что Максим написал в описание PR.
- `diff.patch` — diff в формате `git diff` (для удобства просмотра «как это выглядит в GitHub»).

### Задание

1. **Прочитай PR полностью.** Минимум 10 минут — не торопись, изучи SQL, посмотри `schema.yml`.
2. **Найди 5 проблем.** Они там есть. Распределение: 2 blocker, 2 nit, 1 question.
3. **Открой `review-template.md`** — там объяснение типов комментариев и шаблоны.
4. **Напиши свои review-комментарии** в файле `my-review.md` (создашь сам в этой папке). Используй формат из шаблона.
5. **Сверь** свои комментарии с `expected-review.md`. Не подсматривай раньше — это будет нечестно к себе.

### Критерии хорошего review-комментария

- Указывает **что** не так (конкретная строка / блок).
- Указывает **почему** это проблема (контекст: производительность? стиль? bug?).
- Предлагает **решение** или хотя бы вариант (а не «исправь»).
- Помечен `blocker:` / `nit:` / `question:` / `suggestion:` — чтобы автор понимал серьёзность.
- Не переходит на личности. Критикуем код, а не человека.

Антипаттерны, которые не пиши:

- «Это плохо» — без объяснения.
- «Перепиши» — без подсказок.
- «А почему ты вообще так сделал???» — переход на эмоции.
- «Это не работает» — без приложенного output / трейса.

## Часть 2. Ты — Author

### Контекст

Ты неделю назад открыл свой PR с новой Python-функцией для расчёта retention. Твой тимлид прислал 4 комментария на review. Часть из них — справедливые, часть — спорные.

В папке `author-response/` лежит:

- `my_code.py` — твой исходный код, который попал на review.
- `review-comments.md` — комментарии тимлида.
- `response-template.md` — шаблоны ответов «agree/will fix», «discuss», «push back».

### Задание

1. **Прочитай свой код** (`my_code.py`) — освежи в памяти, что ты делал.
2. **Прочитай каждый комментарий** из `review-comments.md`.
3. **Для каждого комментария реши:**
   - `agree` — согласен, сейчас пофикшу. Опиши, что именно поменяешь.
   - `discuss` — есть нюанс, нужно обсудить. Задай уточняющий вопрос.
   - `push back` — не согласен. Аргументированно объясни почему. Будь готов к компромиссу.
4. **Напиши свои ответы** в файле `my-response.md` (создашь сам в этой папке).
5. **Сверь** с `expected-response.md`.

### Критерии хорошего ответа author

- **Никогда не молчи** — каждый комментарий должен получить ответ. «Тишина» в треде самое худшее.
- **Если согласен — закоммить fix.** Не пиши «fixed» без коммита.
- **Если не согласен — аргументируй с фактами.** Бенчмарк, ссылка на документацию, бизнес-требование.
- **Не переходи на «я знаю лучше».** «Хороший вопрос. Я подумал об этом так:...» работает в 10 раз лучше, чем «нет, ты не прав».
- **Если зашло в тупик — эскалируй.** «Давай обсудим голосом / на стендапе» — нормальная реакция, лучше чем 20 сообщений в треде.

## Чеклист самопроверки

### Часть 1 (Reviewer)

- [ ] Прочитал PR минимум 10 минут.
- [ ] Создал `my-review.md` с 5+ комментариями.
- [ ] Каждый комментарий помечен тегом (`blocker:` / `nit:` / `question:`).
- [ ] Указано **где** (имя файла + номер строки) и **почему** (контекст проблемы).
- [ ] Сверил с `expected-review.md` — нашёл минимум 4 из 5 проблем.

### Часть 2 (Author)

- [ ] Создал `my-response.md` с ответом на каждый из 4 комментариев.
- [ ] Каждый ответ помечен (`agree` / `discuss` / `push back`).
- [ ] Если `agree` — описано, **что именно** будешь менять.
- [ ] Если `push back` — приведён хотя бы один **факт / ссылка / бенчмарк** в поддержку позиции.
- [ ] Сверил с `expected-response.md`.

## Чему ты научился

- Различать **blocker / nit / question / suggestion** — четыре разных уровня важности комментария.
- Писать review-комментарий по формуле **что -> почему -> как починить**.
- Не молчать на review — у каждого комментария должен быть ответ.
- Push back с фактами, а не с эмоциями.
