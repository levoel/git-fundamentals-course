# Шаблоны для review-комментариев

В этой лабе используем упрощённую версию Conventional Comments (https://conventionalcomments.org/). Каждый комментарий начинается с одного из тегов:

| Тег | Когда использовать | Должен ли автор фиксить? |
|---|---|---|
| `blocker:` | Серьёзный баг, security, breaking change, нарушение бизнес-логики | **Да, до merge.** |
| `suggestion:` | Полезное улучшение (производительность, читаемость) — но можно и в follow-up PR | По решению автора. |
| `nit:` | Мелкая придирка по стилю / именованию. Не блокирует. | Опционально. |
| `question:` | Не понимаю, объясни. Может скрывать blocker, может быть просто уточнением. | Должен ответить. |
| `praise:` | «Классная находка / решение» — да, такое бывает. | Не требует действия. |

## Структура комментария

```
### <tag>: <одна строка summary>

**File:** `path/to/file.sql:L<line>`

<2-5 строк описание проблемы + почему>

**Suggested fix:**
<вариант решения — код, ссылка, описание>
```

## Примеры

### blocker:

```
### blocker: hardcoded date breaks backfills

**File:** `models/fct_orders.sql:L12`

Hardcoded `'2024-01-01'` в WHERE клаусе означает, что при backfill в Q4 2023 модель пропустит данные. Если кто-то запустит `dbt run --vars '{start_date: 2023-10-01}'`, дата всё равно будет 2024-01-01.

**Suggested fix:** заменить на `WHERE order_date >= '{{ var("start_date", "2024-01-01") }}'` и определить переменную в `dbt_project.yml`.
```

### suggestion:

```
### suggestion: avoid SELECT * in CTE for performance

**File:** `models/fct_orders.sql:L9`

`SELECT *` в первом CTE подтягивает все 47 колонок `stg_orders`, из которых используем 4. Это лишняя нагрузка на shuffle, особенно на больших таблицах.

**Suggested fix:** явно выбрать только `order_id`, `user_id`, `order_date`, `amount`.
```

### nit:

```
### nit: inconsistent column naming

**File:** `models/fct_orders.sql:L20`

В одном CTE `userid` (без подчёркивания), в другом — `user_id`. dbt-style гайд команды требует snake_case с подчёркиванием.

**Suggested fix:** переименовать `userid` -> `user_id`.
```

### question:

```
### question: why filter by order_date and not created_at?

**File:** `models/fct_orders.sql:L13`

В таблице `stg_orders` есть и `order_date` (когда заказ оформлен), и `created_at` (когда запись попала в БД). Хочется убедиться, что выбор именно `order_date` — осознанный.

Чем может помочь: short reply про бизнес-логику.
```

### praise:

```
### praise: clean separation of CTEs

**File:** `models/fct_orders.sql`

Структура `orders -> customers -> joined -> aggregated` читается очень понятно. Сохраним этот стиль в следующих моделях.
```

## Что НЕ делать в комментариях

| Антипаттерн | Почему плохо | Как лучше |
|---|---|---|
| «Это плохо» | Нет конкретики | «На строке 12 хардкод даты — при backfill сломается.» |
| «Перепиши» | Нет вектора | «Suggested fix: использовать `{{ var(...) }}`.» |
| «Зачем ты так сделал???» | Эмоция | «question: какая мотивация для такого решения?» |
| «Не работает» | Без воспроизведения | «Не работает на пустом dataframe: `pytest test_x.py` -> AttributeError, трейс ниже.» |
| «Всё переделать» | Демотивирует | Разбить на конкретные пункты + предложить парную сессию. |

## Размер review

- **1 PR = 1 цель.** Если PR на 500 строк и три разные фичи — попроси разбить.
- **Если 30+ комментариев — не оставляй их в треде.** Лучше быстрый созвон.
- **Не пиши комментарий сразу.** Прочитай весь PR, потом второй раз с правкой комментариев. Часто первый «blocker» оказывается «всё ок, я просто не понял контекст».

## Tone

Можно представлять, что **за каждым комментарием стоит живой человек**, который вкладывал в этот PR минимум час времени. Цель review — улучшить код **вместе**, а не показать, что reviewer умнее.

«Я попробовал так и наткнулся на эту проблему» **лучше**, чем «Ты сделал так, это сломано».
